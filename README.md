# PetrAuthDemo — ASP.NET Core MVC + Авториза (OIDC Authorization Code Flow)

## Назначение проекта

Демо-приложение на ASP.NET Core 10 MVC, реализующее интеграцию с сервисом **Авториза** по протоколу OpenID Connect.

**Основной функционал:**
- Вход через Авторизу с поддержкой SSO
- Хранение Refresh Token на сервере (не в куке)
- Автоматическое обновление Access Token до истечения срока
- Запрос UserInfo Endpoint с fallback на ID Token
- Выход из системы с очисткой токенов

## Установка зависимостей

Восстановите NuGet-пакеты:

```bash
dotnet restore
```

Клиентские библиотеки (Bootstrap, jQuery, jQuery Validation) не хранятся в репозитории — они перечислены в `libman.json` и восстанавливаются в папку `wwwroot/lib`. Установите LibMan CLI (один раз на машину) и восстановите библиотеки:

```bash
dotnet tool install -g Microsoft.Web.LibraryManager.Cli
libman restore
```

> Если команда `libman` не распознаётся сразу после установки — перезапустите терминал, чтобы обновился PATH.

## Настройка приложения в Авторизе

### Регистрация приложения

1. Перейдите в панель управления Авторизы
2. Создайте новый проект и приложение в нём
3. Укажите **Redirect URI**: `https://localhost:7094/callback`
4. Тип приложения: **Confidential**

### Получение credentials

Скопируйте из конфигурации приложения:
- **client_id**
- **client_secret**

### Настройка конфигурации

В `appsettings.json` укажите `Authority` и `ClientId`:

```json
"OidcSettings": {
  "Authority": "https://<ваш-сервер-авторизы>/oidc",
  "ClientId": "ваш-client-id",
  "ClientSecret": "SECRET"
}
```

`ClientSecret` рекомендуется хранить через .NET User Secrets, а не в `appsettings.json`:

```bash
dotnet user-secrets init
dotnet user-secrets set "OidcSettings:ClientSecret" "ваш-client-secret"
```

## Кнопки входа

На главной странице для неавторизованного пользователя доступны две кнопки входа:

- **Войти (SSO)** — стандартный вход. Если на сервере Авторизы уже есть активная SSO-сессия, токены выдаются прозрачно, без повторного ввода логина и без экрана согласия.
- **Войти с Refresh** — принудительный вход с параметром `prompt=consent`. Гарантированно показывает экран согласия и создаёт новый грант (а значит и новый Refresh Token), даже если SSO-сессия уже активна. Используется, когда нужно заведомо получить свежий Refresh Token — например после его истечения или отзыва.

## Проверка основных сценариев

### 1. Вход (SSO)

```
1. Откройте https://localhost:7094
2. Нажмите "Войти (SSO)"
3. Введите учётные данные в форме Авторизы (если нет активной SSO-сессии)
4. После успешного входа — редирект на /Home/TokenInfo?fetchUserInfo=true
```

**Ожидаемый результат:** на странице отображаются Access Token, ID Token, Refresh Token (с пометкой «Хранится на сервере»), их JWT Payload, время истечения и данные пользователя. Благодаря параметру `fetchUserInfo=true` запрос к UserInfo выполняется автоматически сразу после входа.

### 2. Автоматическое обновление токенов

Автообновление работает двумя путями.

**По таймеру (на открытой странице):**
```
1. Авторизуйтесь и оставайтесь на странице /Home/TokenInfo
2. Дождитесь, когда таймер обратного отсчёта дойдёт до отметки 5 минут
```
JS-таймер на странице автоматически отправит форму обновления.

**При запросе к серверу (в том числе при запуске приложения):**
```
1. Авторизуйтесь, закройте вкладку (кука сохраняется — она постоянная)
2. Вернитесь в приложение, когда до истечения Access Token осталось < 5 минут
3. Откройте любую страницу
```
Серверный обработчик `OnValidatePrincipal` срабатывает при каждом запросе и обновляет токен в фоне.

**Ожидаемый результат:** токены обновляются автоматически, пользователь не замечает подмены. В консоли приложения появится:
```
Access Token истекает через 5 минут. Запускаем автообновление...
Автообновление успешно. Новый Access Token: eyJhbG...
```

> Если к моменту обновления Refresh Token уже истёк или отозван на сервере, приложение завершает сессию (удаляет токены, гасит куку) и при следующем заходе на защищённую страницу перенаправляет на вход.

### 3. Ручное обновление токенов

```
1. Авторизуйтесь в приложении
2. На странице /Home/TokenInfo нажмите "Обновить токены вручную"
```

**Ожидаемый результат:** Access Token обновлён, страница перезагрузилась с новыми данными, UserInfo запрошен повторно.

> Если Refresh Token не найден или отклонён сервером, приложение перенаправляет на принудительный вход (`ForceLogin`) с экраном согласия. При обрыве связи возвращает на страницу токенов без изменений — можно попробовать снова.

### 4. Запрос UserInfo Endpoint

```
1. Авторизуйтесь в приложении
2. На странице /Home/TokenInfo нажмите "Запросить UserInfo"
```

**Ожидаемый результат:** в блоке «Данные профиля» появляется ответ с данными пользователя (`sub`, `email`, `name` и др.) и время получения.

> **Поведение fallback.** Если UserInfo Endpoint возвращает ошибку (на тестовом сервере Авторизы это происходит регулярно — токены и идентификаторы сессии теряются при перезапуске/переключении экземпляров сервера), приложение читает те же данные профиля из подписанного ID Token. В этом случае ответ начинается со строки `[Источник: ID Token — UserInfo Endpoint вернул ошибку]`, за которой следует текст ошибки и данные из ID Token. Это штатное поведение, а не сбой приложения — данные профиля отображаются в любом случае.

### 5. Выход

```
1. Нажмите "Выйти (<email>)" в навигационной панели
```

**Ожидаемый результат:** Refresh Token удалён из `tokens.json`, кука очищена, пользователь перенаправлен на главную страницу в неавторизованном состоянии.

## Структура репозитория

| Файл | Назначение |
|---|---|
| `Program.cs` | Конфигурация OIDC, middleware, автообновление токенов (`OnValidatePrincipal`) |
| `Controllers/HomeController.cs` | Маршруты: Index, TokenInfo, FetchUserInfo, RefreshTokens, Login, ForceLogin, Logout |
| `Services/TokenStorage.cs` | Хранение Refresh Token в `tokens.json`, синхронизация через SemaphoreSlim |
| `Models/ErrorViewModel.cs` | Модель страницы ошибки |
| `Views/Home/TokenInfo.cshtml` | Отображение токенов, JWT Payload, UserInfo, JS-таймер автообновления |
| `Views/Shared/_UserInfoPartial.cshtml` | Partial view для AJAX-запроса UserInfo |
| `appsettings.json` | Конфигурация OIDC (Authority, ClientId; ClientSecret — плейсхолдер) |
| `libman.json` | Список клиентских библиотек для восстановления через LibMan |
| `tokens.json` | Серверное хранилище Refresh Token (создаётся автоматически, в репозиторий не попадает) |
